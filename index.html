Definition
What is Vibe Coding?

The idea

Vibe coding is the practice of building software by directing an AI model in plain English ‚Äî describing what you want, reviewing what it builds, and iterating conversationally. You are the architect and decision-maker. The AI does the typing.


It's not a shortcut that produces bad code. Done well, it produces professional, production-ready software ‚Äî at a fraction of the time and skill barrier traditionally required.

What you still control

  • What to build and why ‚Äî the product vision
  • Architectural decisions ‚Äî database design, service choices
  • Scope management ‚Äî what's in v1 vs v2
  • Quality bar ‚Äî reviewing what AI produces
  • User experience ‚Äî what feels right
  • Business logic ‚Äî pricing, flows, edge cases

What the AI handles

  • Writing the actual code files
  • Installing and configuring packages
  • Debugging error messages
  • Implementing known patterns (auth, forms, queries)
  • Repetitive but precise work (security policies, config)
  • Explaining what it just built

The honest caveat

Vibe coding amplifies your judgment ‚Äî it doesn't replace it. A vague prompt produces vague code. A thoughtful builder who understands the problem, even without deep technical skills, will consistently outperform a passive one who just accepts whatever the AI generates.


The skill is knowing what to ask for, not knowing how to write it.

"The bottleneck was never typing speed. It was always knowing what to build next."

‚Äî Common realization among first-time vibe coders
02
Infrastructure
The Modern Stack
Frontend + Routing
Next.js
React-based framework that handles both the user interface and server-side logic in a single codebase. The industry default for modern web apps.
Why: Full-stack in one repo. Integrates with everything. Vercel (same company) deploys it in seconds.
Auth + Database + Storage
Supabase
Open-source Firebase alternative. Gives you user authentication, a Postgres database, file storage, and real-time data sync ‚Äî all in one service.
Why: Replaces 3-4 separate services. Real SQL database means your data can grow. Generous free tier.
Hosting + Deployment
Vercel
Push code to GitHub. Vercel detects the change and deploys automatically ‚Äî usually in under 60 seconds. Global CDN included.
Why: Zero DevOps. The pipeline (code ‚Üí deploy) is the most important habit to build early.
Styling
Tailwind CSS
Utility-first CSS framework. Instead of writing custom stylesheets, you apply pre-built classes directly in your HTML. Faster, more consistent.
Why: AI writes Tailwind exceptionally well. Readable, maintainable, and design-system-ready from day one.
Payments (when needed)
Stripe
The standard for payment processing. Handles card charges, subscriptions, marketplace splits, payouts, and tax forms (1099s) automatically.
Why: Non-negotiable when money moves. Stripe Connect specifically handles marketplace payment routing.
Version Control
Git + GitHub
Git tracks every change you make. GitHub stores your code in the cloud. Together they're your undo button, your backup, and your deployment trigger.
Why: Commit working code constantly. This is the safety net that lets you experiment without fear.

Communication
The Art of the Prompt

The quality of what you build is directly proportional to the quality of what you ask for. Vague in, vague out. Specific in, professional out. This is the skill worth developing.

‚úó Weak prompts
"Build me the login page"
"Make it look better"
"Add a booking system"
"Fix the bug"
"Build the whole app"
‚úì Strong prompts
"Build an email + password login form that redirects to /dashboard on success and shows an inline error on failure"
"Increase spacing between cards to 24px and change the heading font weight to 700"
"Create a form that takes a date and a note, writes a row to the bookings table with status pending"
"I'm seeing 'permission denied on table bookings' ‚Äî here's the full error: [paste error]"
"Build only the task creation form today. We'll add the toggle and history in the next session."
Anatomy of a great prompt ‚Äî every strong prompt has these four parts
Context We have a tasks table in Supabase with columns: id, name, whose_turn_id, last_done_at
Goal Build a button that, when tapped, updates last_done_at to now and flips whose_turn_id to the other household member
Scope Just the button and its click handler. Don't build the full task list or history view yet.
Format Add a comment explaining what the flip logic does. Tell me what file to put this in.

06
Principles
Rules That Actually Matter
1
Ship phase one before thinking about phase two
The most common failure mode in vibe coding isn't bad code ‚Äî it's scope creep. Every new idea feels important in the moment. The discipline is ruthlessly asking: "Is this phase one?" If the answer is no, write it down and keep building. A working narrow product is infinitely more valuable than an unfinished broad one. Every feature you add before shipping is a bet that users want it. Every feature you add after shipping is informed by evidence.
‚Üí Define phase one on day one. Write it down. Show it to someone. Make them hold you to it.
2
Commit to git every time something works
Git is your undo button. If you build for 3 hours without committing and then break something, you have no safe place to return to. The habit: every time a feature works, before starting the next one, run git commit -m "what just started working".
‚Üí Working auth ‚Üí commit. Form submits ‚Üí commit. Bug fixed ‚Üí commit.
3
Understand before continuing
When Claude writes something you don't understand at all, stop and ask it to explain. Don't just keep building on code you can't reason about. You'll hit a bug eventually, and if you don't understand what the code does, you can't describe the problem clearly enough for anyone ‚Äî human or AI ‚Äî to help you fix it.
‚Üí "Before we continue, explain what this useEffect is doing and why."
4
Never commit secrets to GitHub
Your API keys, database passwords, and Stripe keys live in a .env.local file that must never be committed to version control. Add it to .gitignore on day one. A committed secret is a compromised secret ‚Äî bots scrape GitHub for them within minutes.
‚Üí Check .gitignore before your first commit. Every project, every time.
5
Start sessions with context
Claude doesn't remember your last conversation unless you're in a Project with good instructions. Starting a session with "here's where we are and here's today's goal" takes 30 seconds and saves 20 minutes of misaligned output. Treat every Claude Code session like onboarding a new contractor: brief them before they touch anything.
‚Üí "We're building X. Here's what's done. Today's goal is only Y."
6
Build the hard thing on a simpler project first
Every complex app contains patterns that can be learned in isolation. Payments, auth, file uploads, real-time sync ‚Äî each of these can be built standalone in a day. Master the pattern on a low-stakes project before wiring it into your main product. Your confidence and your code quality will both be higher.
‚Üí Build a "buy me a coffee" page before a full marketplace checkout.
